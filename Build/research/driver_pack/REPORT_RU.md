# Исследование воспроизводимости Hrust-блоков драйверов WildDOS

## Итог

Четыре runtime-образа из production ASM собираются побайтно точно. Найден локальный
командный упаковщик MHMT с поддержкой `Hrust1.x`, но его результат **не совпадает**
с четырьмя историческими блоками `WDOSFC35.CCB`.

Точный командный режим локально не найден. Исторический упаковщик
`exe/WC/MENU/Hrust v1.3.spg` присутствует, но это интерактивная TR-DOS/SPG-программа,
а не Windows CLI. В `source` уже лежат точные исторические результаты для NEMO и
SMUC; для актуальных SDZC и DSDNGS соответствующих точных `.CPD` нет.

Production-файлы и `build.ps1` в ходе исследования не изменялись. Все полученные
файлы находятся только в `Build/research/driver_pack`.

## Фактический 12-байтовый заголовок DEHR2

Формат доказан кодом `source/DEHR2.ASM` и четырьмя reference-блоками:

| Смещение | Размер | Значение |
|---:|---:|---|
| `+0` | 2 | ASCII `HR` (`48 52`); DEHR2 сигнатуру не проверяет |
| `+2` | 2 | длина распакованного runtime, little-endian |
| `+4` | 2 | полная длина packed-блока с заголовком, little-endian |
| `+6` | 6 | последние 6 байт распакованного runtime |
| `+12` | переменно | Hrust 1.x, 16-битные little-endian слова битового потока |

Поле `+4` DEHR2 применяет только при перемещении перекрывающегося входа через
`LDDR`. Во время декодирования проверки физической границы packed-блока нет.
Контрольной суммы и кода ошибки также нет.

Reference-заголовки:

| Драйвер | 12 байт заголовка |
|---|---|
| NEMO | `48 52 BF 03 E7 01 24 15 20 DC 18 D0` |
| SDZC | `48 52 8C 04 A4 02 39 3B 22 10 3B C9` |
| DSDNGS | `48 52 62 05 32 03 20 FB F1 C1 C9 00` |
| SMUC | `48 52 87 03 77 01 3D C2 51 3B 24 C9` |

## Проверка MHMT

Команда упаковки совместимого потока:

```powershell
& 'C:\Users\Администратор\Desktop\WC\ZiFi\_spg\mhmt.exe' -hst -zxh INPUT.bin OUTPUT.hst
```

Параметры по выводу MHMT: Hrust1.x, optimal coding, 16-битные little-endian слова,
ZX header. Однако MHMT записывает в `+4..+5` **повтор длины распакованного файла**,
а не packed length:

| Драйвер | Reference `+4` | MHMT `+4` |
|---|---:|---:|
| NEMO | `#01E7` | `#03BF` |
| SDZC | `#02A4` | `#048C` |
| DSDNGS | `#0332` | `#0562` |
| SMUC | `#0177` | `#0387` |

Поэтому сырой `-zxh`-результат нельзя подставлять в перекрывающуюся процедуру
DEHR2 без исправления `+4..+5` на фактическую длину файла. После такой правки все
четыре MHMT-блока распаковываются в точные runtime-образы, то есть формат потока
совместим. Но packed-байты не exact:

| Драйвер | Reference | MHMT optimal | Разница длины | Первый разный байт потока |
|---|---:|---:|---:|---|
| NEMO | 487 | 482 | -5 | `+0C`: reference `D1`, MHMT `D0` |
| SDZC | 676 | 666 | -10 | `+0C`: reference `C5`, MHMT `C2` |
| DSDNGS | 818 | 808 | -10 | `+12`: reference `5D`, MHMT `2D` |
| SMUC | 375 | 369 | -6 | `+12`: reference `5D`, MHMT `2D` |

Примечание: в последнем столбце `+12` означает шестнадцатеричное смещение
`0x12`, а не начало потока `+0x0C`.

Дополнительные проверки:

- `-maxwin512` дал NEMO ту же длину 487, но другой SHA-256
  `DA76E05243B1F04EC642E0621D84088683805EF6B5C80F4E68340C9DC3C053CA`;
  совпадение длины случайное.
- Окна 256/512/1024 не дали exact ни для одного драйвера.
- `-g` для Hrust заканчивается сообщением
  `greedy coding not supported`; получить алгоритм старого Hrust этим ключом
  нельзя.

## Граничный случай DSDNGS

При строгой распаковке файла ровно в заявленные 818 байт MHMT формирует 1372
байта и завершает работу ошибкой `Can't get bits from input stream!`. Это ровно
`1378 - 6`: все данные до вынесенного в заголовок шестибайтного хвоста уже готовы,
но терминатор ещё не прочитан.

Если после 818 байт добавить первые два байта следующего вплотную блока SMUC
(`48 52`, то есть `HR`), MHMT успешно завершает распаковку и выдаёт exact runtime
длиной 1378. Следовательно, исторический DSDNGS-поток использует разрешённое
старым Z80-декодером чтение следующего 16-битного слова за declared packed length.
В штатном `WDOSFC35.CCB` это работает, поскольку блоки хранятся вплотную, а DEHR2
не контролирует границу потока. Изолированный DSDNGS-блок не является строгим
самодостаточным MHMT-файлом без двух байт look-ahead.

## Причина невоспроизводимости exact

MHMT реализует совместимый, но другой оптимизатор/эмиттер Hrust 1.x и иной вариант
ZX-заголовка. Его поток уже с первого кодового слова выбирает другие токены и
получается на 5–10 байт короче reference. Перебор документированных параметров
MHMT это различие не устраняет.

Наиболее вероятный источник reference — интерактивный `Hrust v1.3.spg`: он лежит
в составе того же WC, а два сохранённых `.CPD` побайтно равны reference. Это
сильное свидетельство, но не доказанная команда сборки: локального журнала запуска,
исходников компрессора или автоматизируемого CLI для SPG нет. Поэтому заявлять
exact-воспроизводимость всех четырёх packed-блоков нельзя; доказана только точная
воспроизводимость runtime и совместимая перепаковка MHMT после правки `+4`.
