# CORE32: единый API и текущее поведение

Этот документ является цельным справочником по файловому ядру CORE32 в
улучшенной сборке `Wild Commander v1.10i`. Он описывает:

- официальный файловый API плагинов Wild Commander;
- внутреннюю таблицу WildDOSFileCore по адресу `#4000`;
- новый API 76 `APPEND` и все известные результаты его работы;
- исправленное поведение FAT32, каталогов, LFN, FSInfo и потоков;
- ограничения, которые текущая сборка намеренно не скрывает.

Документ соответствует `exe/boot.$C` размером 31761 байт и SHA-256
`f1234d96552c5b61f4bf3b091472279d5fd24983fd9a8cea5ff1bd82323ee498`.
Журнал доказательств, хэши остальных артефактов и история дефектов находятся в
[`CORE32_audit.md`](CORE32_audit.md).

В обозначениях ниже `#` означает шестнадцатеричное число, `LE32` — четыре
байта little-endian, `Z/NZ` и `CF` — флаги Z80 после возврата.

## 1. Какой интерфейс вызывать

| Вызывающий код | Точка вызова | Как передаётся функция | Назначение |
|---|---:|---|---|
| Обычный плагин WC | `WLD=#6006` | номер `0..86` в `A` | Официальный и рекомендуемый интерфейс |
| Код менеджера/тесно связанный модуль | `#4000+offset` | прямой `CALL` в 31-элементную таблицу | Внутренний ABI CORE32 |
| Само ядро CORE32 | `EXTENSION_GATE=#5123` | приватный байт ID после `CALL` | Деталь реализации, не API плагинов |

Плагин должен вызывать `#6006`. Сам диспетчер только выбирает адрес из таблицы
по номеру в `A`; подключение рабочей файловой страницы, преобразование
состояния менеджера и восстановление отображения выполняют соответствующие
файловые `X*`-обёртки. Прямые вызовы `#40xx` требуют ручного управления
страницами и не являются безопасной заменой публичному API.

Общий шаблон плагинного вызова:

```asm
        ; Остальные входные регистры подготовлены согласно функции.
        LD A,api_number
        CALL #6006
        ; Проверить документированные флаги до любых команд, меняющих F.
```

Если функции нужен входной `A`, плагин передаёт его через `A'`: основной `A`
занят номером API, а диспетчер выполняет `EX AF,AF'` перед целевой процедурой.

Важно: `#6004` — флаг Esc менеджера плагинов. Это не внутренний `ABT` CORE32,
который находится в рабочей файловой странице по адресу `#384A`.

## 2. Модель памяти и состояния

| Диапазон/адрес | Содержимое |
|---:|---|
| `#0000..#1FFF` | Буферы перечисления каталога |
| `#3000..#31FF` | Сектор FAT (`SECBU`) |
| `#3200..#33FF` | Основной секторный буфер (`LOBU`) |
| `#3400..#35FF` | Второй секторный буфер (`LOBU2`) |
| `#37F8..#37FE` | Состояние итератора каталога |
| `#3800` | Таблица и код выбранного драйвера |
| `#3848..#38CC` | Поток, `EOC`, `ABT`, `ENTRY`, BPB/FAT и текущий каталог |
| `#4000..#5BF1` | Установленный код CORE32; `END=#5BF2` |
| страница `#E0`, `#C000..#CC39` | Приватный runtime `CORE32_EXT` во время шлюза |

CORE32 рассчитан на FAT32 с 512-байтовыми секторами. Его секторные буферы,
поток, текущий каталог, `ENTRY`, обработчик `NW0` и контекст APPEND глобальны.
API не является повторно-входимым и не поддерживает параллельные файловые
операции.

Перед прямым вызовом `#40xx` рабочая файловая страница должна быть подключена к
окну `#0000`. В текущем WC это делает пара `CORE_PAGE_IN=#6B7A` и
`CORE_PAGE_OUT=#6B8B`. Пара одноуровневая: вложенный вход перезаписывает
единственный сохранённый номер страницы. Все выходы, включая ошибочные, должны
быть сбалансированы.

Обычному плагину нельзя вызывать эту пару напрямую. Файловые `X*`-обёртки API
48–52, 54–63 и 72–76 выполняют переключение сами. API 53 является прямым
пустым `FEX` и страниц не меняет. Приватные ID из
[`CORE32_EXT_API.ASM`](CORE32_EXT_API.ASM), адрес страницы `#E0` и внутренние
процедуры `CORE32_EXT` не входят в стабильный ABI.

## 3. Публичный файловый API плагинов WC

Это фактическая файловая часть таблицы `PLUGIN_API_TABLE` из
[`MD20.ASM`](MD20.ASM). Десятичный номер передаётся в `A` при вызове `#6006`.
Номера плагинного API не совпадают с индексами внутренней таблицы CORE32.

| API | Имя | Основной контракт |
|---:|---|---|
| `48 (#30)` | `LOAD512` | `HL` — буфер, `B` — число 512-байтовых блоков; читает текущий поток, возвращает новый `HL` и потоковый статус |
| `49 (#31)` | `SAVE512` | Как `LOAD512`, но записывает существующую цепочку; файл не расширяет и размер не меняет |
| `50 (#32)` | `GIPAGPL` | Позиционирует поток по сохранённому менеджером кластеру |
| `51 (#33)` | `TENTRY` | `DE` — буфер; копирует в него сохранённую 32-байтовую FAT32 ENTRY |
| `52 (#34)` | `CHTOSEP` | `DE` — начало списка LBA32, `BC` — исключающая верхняя граница; раскладывает сохранённую цепочку в сектора |
| `53 (#35)` | reserved | Пустой обработчик; это не APPEND |
| `54 (#36)` | `TMRKFL` | Найти/отметить объект через внутреннее состояние менеджера |
| `55 (#37)` | reserved | Пустой обработчик |
| `56 (#38)` | `ADIR` | Открыть каталог, описанный сохранённым состоянием менеджера |
| `57 (#39)` | `STREAM` | Управление потоком: `D` — поток, `B` — устройство, `C` — раздел; `BC=#FFFF` включает поток `D` |
| `58 (#3A)` | `NXTETY2` | `DE` — выходной буфер, входная маска в `A'`; получить следующую запись каталога |
| `59 (#3B)` | `FENTRY` | `HL` → `[flag,name,0]`; `Z` — не найдено, `NZ` — найдено; сохраняет метаданные и контекст APPEND |
| `60 (#3C)` | `LOAD256` | Потоковое чтение через 256-байтовый обработчик драйвера; поддержка зависит от драйвера |
| `61 (#3D)` | `LOADNONE` | Линейно продвигает поток без копирования данных |
| `62 (#3E)` | `GFILE` | После успешного `FENTRY` позиционирует поток на начало найденного файла |
| `63 (#3F)` | `GDIR` | После успешного `FENTRY` делает найденный каталог активным |
| `72 (#48)` | `MKFILE` | `HL` → `[attr,size_le32,name,0]`; `Z` — создан, `NZ` — ошибка |
| `73 (#49)` | `MKDIR` | `HL` → имя с нулём; `Z` — создан, `NZ` — ошибка |
| `74 (#4A)` | `RENAME` | `HL` → старое `[flag,name,0]`, `DE` → новое имя с нулём |
| `75 (#4B)` | `DELFL` | `HL` → `[flag,name,0]`; удаляет запись и цепочку, но не защищает непустой каталог |
| `76 (#4C)` | `APPEND` | `HL` — буфер, `BC=0..512`; дописывает в конец последнего успешно найденного файла |

Формат поиска `FENTRY`/`DELFL`: первый байт `#00` означает файл, `#10` —
каталог, затем идёт имя и завершающий ноль. Текущий валидатор принимает имя до
255 байт, отвергает 256 байт, управляющие символы и `"*/:<>?\|`. Конечные
пробелы и точки отрезаются с изменением входного буфера. Любой пробел заставляет
создать LFN; начальные пробелы при поиске LFN игнорируются симметрично, а
внутренние остаются значимыми.

Сохранившиеся C-копии `wc_api.h` содержат исторические неточные подписи:
`API 58` является `NXTETY2`, а создание файла выполняет `API 72`; `LOAD256` и
`LOADNONE` подключены, хотя были помечены `reserved ???`. Фактическими
источниками истины являются таблица `MD20.ASM` и этот документ.

## 4. API 76 APPEND

### 4.1 Назначение и выбор файла

`APPEND` дописывает один блок в файл, найденный последним успешным `FENTRY`.
Официальный вызов плагина — `A=76`, `CALL #6006`; внутренний вход CORE32 —
`+#27/#4027`.

Обязательная последовательность для ненулевой записи:

1. Вызвать API 59 `FENTRY` для нужного файла.
2. Немедленно проверить `NZ`.
3. Подготовить `HL/BC` и вызвать API 76.
4. После каждого APPEND немедленно проверить `Z/NZ`.

`GFILE` для APPEND не нужен. Последовательные успешные вызовы используют
кэшированное положение EOF и не обходят FAT от начала. Повторный успешный
`FENTRY` того же файла заново вычисляет EOF из 32-битного размера и цепочки.

Контекст один на всё ядро и заменяется каждым успешным `FENTRY`. Неуспешный
`FENTRY` старый контекст не очищает. Если проигнорировать его `Z`, последующий
APPEND способен дописать предыдущий успешно найденный файл.

### 4.2 Вход и результат

| Регистр/флаг | Контракт |
|---|---|
| `HL` | Начало исходного буфера |
| `BC` | Число байт `0..512` |
| Диапазон | Весь полуинтервал `[HL,HL+BC)` должен лежать в `#4000..#C000` |
| Успех | `Z`, `A=0`; при `BC>0` `HL` указывает сразу за использованными данными |
| Ошибка | `NZ`, `A` содержит код; число фактически изменённых байт не сообщается |

Последний читаемый байт не может находиться выше `#BFFF`; конец ровно
`#C000` допустим. Переполнение 16-битного адреса является ошибкой. Текущая
реализация сохраняет `DE`; `BC` и незаявленные регистры следует считать
разрушенными. `CF` результатом APPEND не является и нормализуется в ноль.

Особый случай: `BC=0` безусловно возвращает `Z/A=0`, не проверяя `HL`, контекст
`FENTRY`, тип объекта или состояние цепочки. Это только пустая операция, а не
способ проверить готовность файла.

### 4.3 Порядок записи

Для ненулевого блока CORE32 выполняет:

1. Только на первом APPEND после успешного `FENTRY`: полную сверку SFN,
   проверку типа/размера, обход FAT до EOF и создание кэшированного контекста.
2. Вычисление нового 32-битного размера.
3. Цикл по частям блока: если очередная часть требует нового кластера — сначала
   выделение отдельного EOC-кластера, ещё не связанного со старым хвостом;
   затем read-modify-write существующего сектора либо обнуление нового и запись
   этой части данных. При пересечении границы старый неполный сектор может быть
   записан до выделения кластера для остатка блока.
4. Во время commit — связывание старого хвоста с уже выделенным кластером.
5. Обновление FSInfo.
6. Полную сверку сохранённой SFN-записи с каталогом на диске.
7. Последней операцией — запись первого кластера и нового 32-битного размера.

При `BC<=512` один вызов пересекает не более одной границы кластера и выделяет
не более одного нового кластера. Пустой файл с `cluster=0` получает первый
кластер только при первой ненулевой записи. Время изменения и атрибуты файла
APPEND не обновляет; отдельной проверки атрибута read-only нет.

После первого успешного блока последующие вызовы доверяют кэшированному EOF до
фазы commit и не выполняют предварительную сверку SFN перед записью данных.
Если другой код изменит запись между блоками, commit обнаружит это только
после записи очередных данных и вернёт `#23`. Контекст не является блокировкой.

Этот порядок публикует новый размер последним, но не является атомарной
транзакцией и не обеспечивает устойчивость к потере питания.

### 4.4 Откат

Если был выделен новый кластер и последующая стадия завершилась ошибкой,
APPEND пытается восстановить прежнюю EOC-ссылку, освободить новый кластер и
обновить FSInfo. При успешном откате возвращается исходная ошибка; при ошибке
компенсации — `APPEND_ERR_ROLLBACK`.

Откат не восстанавливает уже записанные секторы данных. При сбое записи
каталога, потере питания или неоднозначном ответе устройства нельзя по одному
коду возврата определить, был ли новый размер опубликован. Журнала и ключа
идемпотентности API не предоставляет.

### 4.5 Коды ошибок

| Код | Имя | Значение |
|---:|---|---|
| `#20` | `APPEND_ERR_CONTEXT` | Нет действительного контекста успешного `FENTRY` или положение SFN недопустимо |
| `#21` | `APPEND_ERR_BUFFER` | Буфер вне `#4000..#BFFF`, пересекает `#C000` или адрес переполняется |
| `#22` | `APPEND_ERR_LENGTH` | `BC>512` |
| `#23` | `APPEND_ERR_ENTRY_CHANGED` | 32-байтовая SFN-запись изменилась после `FENTRY` |
| `#24` | `APPEND_ERR_SIZE_OVERFLOW` | Новый размер превысил `#FFFFFFFF` |
| `#25` | `APPEND_ERR_BAD_CHAIN` | Недопустимый кластер/ссылка, преждевременный EOC либо любой отказ `CURIT`, включая границу FAT или чтение; это не обязательно доказанная порча на диске |
| `#26` | `APPEND_ERR_IS_DIRECTORY` | Контекст относится к каталогу |
| `#27` | `APPEND_ERR_INTERNAL` | Внутренне противоречивое состояние выделения/фиксации |
| `#2A` | `APPEND_ERR_ROLLBACK` | Компенсирующий откат FAT/FSInfo не завершился |

Список не исчерпывает все `NZ`: наружу могут пройти `#10` от `MKSG`
(нет места/ошибка выделения), `#01` и иной ненулевой код `ABT` драйвера.
Вызывающий код обязан считать ошибкой любое `NZ`, а не только известные
константы.

Слепой повтор блока после ошибки запрещён. До записи носителя гарантированно
возникают `#21/#22/#24`: после `#21/#22` можно исправить аргументы, а после
`#24` — уменьшить положительный блок, только если он ещё помещается в
`#FFFFFFFF`. Во всех неоднозначных случаях нужно заново выполнить успешный
`FENTRY`, проверить фактический размер и при необходимости содержимое. После
`#2A` состояние FAT считается неопределённым до проверки файловой системы.

### 4.6 Минимальный пример

```asm
WLD             EQU #6006

        LD HL,FILE_QUERY
        LD A,59                  ; FENTRY
        CALL WLD
        JR Z,.not_found          ; обязательно: старый APPEND-контекст жив

        LD HL,DATA_BUFFER
        LD BC,DATA_SIZE          ; 1..512
        LD A,76                  ; APPEND
        CALL WLD
        JR NZ,.append_error      ; A содержит код; блок вслепую не повторять
        ; успех

FILE_QUERY:
        DB #00,"LOG.BIN",0

DATA_BUFFER:
        DS 512
DATA_SIZE       EQU 512
```

`DATA_BUFFER` в реальном плагине должен целиком находиться в разрешённом
runtime-диапазоне `#4000..#BFFF`, а не только иметь подходящий файловый offset.

## 5. Внутренняя таблица CORE32

`START=#4000` содержит 31 стабильный трёхбайтовый `JP`. Адрес входа равен
`#4000+offset`; таблица предназначена для WC/WildDOS и требует подключённой
рабочей страницы `#0000`.

| Offset / адрес | Имя | Фактический контракт |
|---:|---|---|
| `+#00 / #4000` | reserved | Пустой `RET`; `SELDEV` находится в драйвере `#3800` |
| `+#03 / #4003` | `DEVINI` | Инициализировать установленный драйвер; обычно `Z` — успех |
| `+#06 / #4006` | reserved | Пустой `RET`; `REINI` находится в драйвере `#3806` |
| `+#09 / #4009` | `HDD` | Найти FAT32 и заполнить BPB/FAT-параметры; `Z` — успех, `NZ` — не найдено/ошибка |
| `+#0C / #400C` | `RDD` | `DE:HL=LBA`, `BC=буфер`, `A=секторы`; `Z/NZ` по `ABT` |
| `+#0F / #400F` | `SDD` | Как `RDD`, но запись |
| `+#12 / #4012` | `GIPAG` | `HL` → `cluster32`, ноль означает root; позиционировать на первый сектор |
| `+#15 / #4015` | `LOAD512` | `HL=буфер`, `B=секторы`; потоковое чтение и статус EOC/error |
| `+#18 / #4018` | `SAVE512` | Как `LOAD512`, но запись существующей цепочки без изменения размера |
| `+#1B / #401B` | `DOS_SWP` | `A=0..6`: Nemo master, SD-ZC, SD-NGS, SMUC master/slave, Nemo slave, второй ZC |
| `+#1E / #401E` | `GLSTCAT` | Историческое имя: фактически копирует `cluster32` из `HL` в `LSTCAT` и сбрасывает итератор |
| `+#21 / #4021` | `TLSTCAT` | Историческое имя: фактически копирует текущий `LSTCAT` в четыре байта по `DE` |
| `+#24 / #4024` | `SRHFCL` | Найти свободный кластер от `FSTFRC`; успех `DE:HL/Z`, ошибка `CF=1` |
| `+#27 / #4027` | `APPEND` | Внутренний вход API 76; контракт раздела 4 |
| `+#2A / #402A` | `MKSG` | `DE:HL=размер`; создать цепочку, первый кластер вернуть через `FCTS` |
| `+#2D / #402D` | reserved | Пустой `RET` |
| `+#30 / #4030` | `RFRH` | Проверить и обновить FSInfo; `Z` — успех, `NZ` — сигнатуры/I/O |
| `+#33 / #4033` | `GENTRY` | Фактически копирует 32 байта из `HL` во внутреннюю `ENTRY` |
| `+#36 / #4036` | `TENTRY` | Копирует `ENTRY` в `DE`, сохраняет `A/S/Z/C` и фиксирует контекст APPEND; `LDIR` меняет остальные флаги |
| `+#39 / #4039` | `MKFILE` | `HL` → `[attr,size_le32,name,0]`; `Z` — успех |
| `+#3C / #403C` | `MKDIR` | `HL` → `name,0`; создаёт `.`/`..`, `Z` — успех |
| `+#3F / #403F` | `DELFL` | `HL` → `[flag,name,0]`; удалить запись и цепочку |
| `+#42 / #4042` | `RENAME` | `HL` → старое `[flag,name,0]`, `DE` → новое `name,0` |
| `+#45 / #4045` | `DLSG` | `HL` → `cluster32`; освободить цепочку, стабильного прикладного статуса нет |
| `+#48 / #4048` | `CHTOSE` | `HL` → `cluster32`, `DE` — список LBA32, `BC` — конец буфера |
| `+#4B / #404B` | reserved | Пустой `RET` |
| `+#4E / #404E` | `SRHDRN` | `HL` → `[flag,name 1..255,0]`; `NZ` — найдено, `ENTRY` заполнена |
| `+#51 / #4051` | `LOAD256` | Поток через драйверный 256-байтовый обработчик; не все драйверы реализуют его |
| `+#54 / #4054` | `LOADNON` | `B` — число секторов; пройти поток без копирования |
| `+#57 / #4057` | `NXTETY` | `DE` — выход полной записи каталога |
| `+#5A / #405A` | `NXTETY2` | `DE` — выход, `A` — маска полей и типа записи |

Направления `GLSTCAT/TLSTCAT` и `GENTRY/TENTRY` приведены по фактическим
`LDIR`, а не по исторической букве в имени.

Для `NXTETY2` маска имеет следующий вид:

- бит 7 — выдавать только короткое имя;
- бит 4 — добавить время создания;
- бит 3 — добавить дату создания;
- бит 2 — добавить размер;
- биты 1..0: `0` — любой объект, `1` — файл, `2` — каталог.

Прямые `SRHDRN/NXTETY/NXTETY2` не дают отдельного надёжного прикладного кода
для каждой ошибки чтения: часть отказов исторически выглядит как «не найдено»
или конец перечисления. Публичные обёртки сохраняют совместимость с этим ABI.

## 6. Контракт потоков

Для нормального потока и явно возвращённой драйвером ошибки `ABT` у
`LOAD512`, `SAVE512`, `LOAD256` и `LOADNON` действует результат:

| Состояние | `A` | `CF` | Значение |
|---|---:|---:|---|
| Запрос завершён внутри цепочки | `#00` | `0` | Можно продолжать поток |
| Достигнут физический EOC FAT | `#0F` | `0` | Штатный конец цепочки |
| Возвращённая ошибка I/O/FAT | `#FF` | `1` | Операцию остановить |

`#0F` означает конец FAT-цепочки, а не отдельный 32-битный логический offset.
При нескольких секторах на кластер завершённый короткий запрос может вернуть
`A=0`, если EOC ещё не читался.

Известное исключение внутреннего пути: при переходе к следующему кластеру
последовательность `CALL CURIT,GIPAG` пока не проверяет `CF` от `CURIT`
отдельно до `GIPAG`. Поэтому отказ чтения/границы FAT именно на этом переходе
не доказан как обязательный `#FF/CF=1`. Таблица выше является контрактом
нормализованных путей, но не заменяет проверку размера и состояния носителя.

`SAVE512` намеренно сохраняет историческую семантику: он пишет только уже
существующую цепочку, не выделяет новый кластер и не обновляет размер SFN.
Для роста файла используется только APPEND.

Низкоуровневые `RDD/SDD` возвращают `Z/NZ` по внутреннему `ABT`; их `CF` не
является потоковым результатом. Старый драйвер, который зависает до возврата
или не выставляет `ABT`, не может быть превращён высокоуровневой обёрткой в
управляемую ошибку.

## 7. Исправленное поведение FAT32

### FAT и кластеры

- При включённом зеркалировании изменения записываются во все FAT.
- При `BPB_ExtFlags.bit7=1` изменяется и читается только выбранная активная FAT.
- Служебный старший полубайт FAT32 маскируется при чтении и сохраняется при записи.
- Обычный кластер, EOC `#0FFFFFF8..#0FFFFFFF` и недопустимый диапазон
  `#0FFFFFF0..#0FFFFFF7` различаются явно.
- Доступ к FAT ограничен полным 32-битным `BFTSZ`; поиск свободного кластера
  корректно переходит между секторами и один раз оборачивается к кластеру 2.
- Нулевой файл создаётся с `size=0`, `cluster=0` и без выделенной цепочки.

### FSInfo

- Перед записью проверяются lead, structure и trail signatures.
- `FSI_Free_Count` записывается как `#FFFFFFFF` («неизвестно, пересчитать»).
- `FSI_Nxt_Free` получает безопасную подсказку `FSTFRC` либо `#FFFFFFFF`.
- При плохих сигнатурах сектор не перезаписывается и возвращается `NZ`.

### Каталоги, SFN и LFN

- Удаление LFN проверяет sequence/checksum и гасит до 20 связанных LFN-слотов
  через максимум три секторных буфера; SFN записывается последней.
- Имя длиной 255 байт принимается, 256 — отвергается.
- 255-байтовая LFN может быть найдена и удалена через три сектора каталога.
- Пробел в имени принудительно выбирает LFN; начальные пробелы поиска
  нормализуются с обеих сторон, внутренние пробелы остаются значимыми.
- Полностью заполненный каталог без маркера `#00` корректно заканчивает поиск
  по EOC вместо повторного разбора устаревшего буфера.
- `MKDIR` прекращается при ошибке записи и зануляет хвост выделенного кластера.

### Ошибки и монтирование

- Высокоуровневые чтения/записи используют единый `ABT` и прекращают операцию
  после возвращённой ошибки драйвера.
- `LOAD256/LOADNON` временно устанавливают свой `NW0` и восстанавливают
  прежний обработчик с сохранением `AF/CF`.
- Прямой `LOAD512` нормализует `NW0` к `SAFE_RDDSE`; `SAVE512` временно ставит
  `SAFE_SDDSE`, а общий эпилог возвращает `SAFE_RDDSE`, не произвольную прежнюю
  цель.
- Страница расширения `#E0` отключается на время PIO-записи Nemo, поэтому
  драйвер видит исходный буфер, а не код расширения.
- Неверный superfloppy BPB возвращает ошибку вместо бесконечного повторного
  чтения сектора 0.

## 8. Ограничения и обязательные предостережения

- Боевой `DR2` использует точный штатный `DOSpSDZC35.CPD`. Его ожидания
  `WTDO/WAIT/WTBY/DRESP` могут быть бесконечными; аналогичный риск есть у
  `DSDNGS`. Физический timeout текущего boot не считается пройденным.
- `DELFL` не запрещает удаление непустого каталога. Кроме того, его
  исторические флаги не позволяют надёжно отличить некоторые ошибки записи от
  «не найдено/удалено». Проверку пустоты должен выполнить вызывающий код.
- Нет `TRUNCATE`, прямого 32-битного `SEEK`, overwrite по offset и атомарной
  замены файла. `LOADNONE` выполняет только линейный проход до 255 секторов за
  вызов.
- `APPEND` принимает не более 512 байт, хранит один глобальный контекст и не
  обновляет timestamp/read-only policy.
- `NW0`, итератор каталога и APPEND нереентерабельны; параллельные или вложенные
  файловые операции не поддерживаются.
- Откат метаданных best-effort. Нет журнала, recovery после потери питания и
  гарантированно атомарной публикации между FAT, FSInfo и каталогом.
- `LOAD256` зависит от драйвера; в текущем SD-NGS его вход `RDD256` пустой.
- Не все физические отказные стадии LFN/FAT/FSInfo/APPEND были инъектированы.
  Отдельно не доказаны полный диск/FAT, реальные очень большие кластеры,
  фрагментированный трёхкластерный каталог и потеря питания.
- Текущий exact-бинарник проверен только в разрешённом патченном Unreal, а не
  на реальном железе. NGS, SMUC и slave-варианты не входят в финальную матрицу.

## 9. Подтверждённое тестовое покрытие

Один и тот же exact `boot.$C` прошёл в патченном Unreal:

- `SPC=1`, mirrored FAT;
- `SPC=1`, active FAT1 без зеркалирования;
- `SPC=8`, mirrored FAT;
- `SPC=8`, active FAT1 без зеркалирования;
- штатные UI-циклы F5/F8 на SD-ZC и между SD-ZC/Nemo.

Во всех четырёх CORE32T-режимах основной результат равен
`stage=8 completed=8`, LFN-результат — `completed=10`. APPEND проверен блоками
`3+509+1+511+512+17+100+400=2053` байта, включая повторный `FENTRY`, переходы
секторов/кластеров, точный payload, пять кластеров и нулевой хвост последнего
сектора.

Также проверены пустой файл, SFN/LFN и пробелы, удаление полного LFN-прогона,
границы FAT и FSInfo, полный каталог без `#00`, имена 255/256, неверный BPB и
возврат потоковой ошибки. Тест штатного бесконечного `WTDO` намеренно
пропускается и не является частью PASS отказного пути.
