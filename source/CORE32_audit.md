## Системная карта верхней памяти

| Владелец | Полуинтервал | Страницы |
|---|---:|---|
| TAP/SNA | `[#A1,#C0)` | `#A1..#BF` |
| Mounter RAM-диск | `[#C0,#E8)` | `#C0..#E7` |
| Improved | `[#E8,#F0)` | `#E8..#EF` |
| Commander и система | `[#F0,#100)` | `#F0..#FF` |

`CORE32_EXT` использует `WC_PHYS_CORE32_EXT_PAGE=#E8`. Прежняя страница `#E0`
входила в RAM-диск Mounter и создавала статическую коллизию. CORE32, TAP/SNA и
оба исходника Mounter теперь подключают общую карту. Сборочные `ASSERT`
проверяют смежность диапазонов и запрещают их пересечение.

Системный перенос закрывает E36 на уровне исходников, сборки и упаковки.
Совместный runtime-сценарий «сохранённый RAM-диск Mounter + файловая операция
CORE32» после переноса на `#E8` ещё не выполнен и остаётся обязательной
аппаратной проверкой.

## MKFILE с нулевым размером

### Текущий статус: исправлено

Публичный `MKFILE(0)` не вызывает `MKSG` и не уводит счётчик кластеров в
`#FFFFFFFF`.

Фактический путь:

1. `MKFILE` загружает 32-битный размер в `DE:HL` и вызывает
   `WDOS_EXT.ID_ALLOCATE_FILE`.
2. `ALLOCATE_FILE` проверяет все четыре байта размера.
3. При `DE:HL=0` процедура записывает ноль в `FCTS` и `FCTS+2`, после чего
   возвращает успех.
4. `MKSG` вызывается только для положительного размера.
5. Каталожная запись пустого файла получает `size=0`, `cluster=0`; FAT-цепочка
   не выделяется.

Это проверяет `TEST_EMPTY_FILE` в `tests/core32_unreal/CORE32T.ASM`: после
`MKFILE` и `TENTRY` тест требует нули в `FstClusHI`, `FstClusLO` и 32-битном
размере. Независимая проверка образа требует те же `size=0` и `cluster=0`.

Прямой внутренний `MKSG` рассчитан только на положительный размер. Вызов
`MKSG(0)` запрещён: его старый цикл действительно превратит нулевой счётчик в
`#FFFFFFFF`. Официальный плагин должен использовать публичный `MKFILE`, а не
обходить его через приватный вход `MKSG`.

## APPEND и потоковая запись

API 76 APPEND добавляет `BC=0..512` байт в конец файла, найденного последним
успешным `FENTRY`. Реализованы:

- частичный последний сектор через read-modify-write;
- выделение и линковка следующего кластера;
- обновление FAT с учётом `BPB_ExtFlags`;
- обновление FSInfo;
- публикация нового 32-битного размера последней операцией;
- повторное открытие файла через `FENTRY` и продолжение записи.

В runtime-базисе проверен файл 2 053 байта блоками
`3+509+1+511+512+17+100+400`, пять кластеров и нулевой хвост последнего сектора.

APPEND не является транзакцией: откат выполняется best-effort, а потеря питания
между записями данных, FAT, FSInfo и каталога полностью не закрыта. Усечение,
атомарная замена и прямой 32-битный seek отсутствуют.

## Реестр E1–E36

Статусы в этой таблице являются единственными статусами ошибок в документе.

| Код | Исходная проблема | Текущий статус | Остаток |
|---|---|---|---|
| E1 | Изменялась только FAT0 | Исправлено и пройдено при mirrored FAT и active FAT1 | Расширять отказные тесты новых операций FAT |
| E2 | После удаления SFN оставались LFN | Исправлено для полного прогона до 20 LFN через три сектора | Инъекция отказа записи каждого сектора |
| E3 | Поиск свободного кластера нарушал переход/границу FAT | Исправлено и пройдено на переходе FAT-сектора | Полный оборот и полностью занятая FAT |
| E4 | Грубая классификация EOC и немаскированный ниббл | Исправлено и пройдено на граничных значениях FAT32 | Реальная цепочка с очень большим допустимым кластером |
| E5 | `FSI_Free_Count` оставался устаревшим | Исправлено для корректного, неизвестного и повреждённого FSInfo | Физический отказ записи относится к E15 |
| E6 | Старый `MKFILE(0)` доходил до `MKSG` и получал `#FFFFFFFF` | Исправлено: ноль обрабатывает `ALLOCATE_FILE`, `MKSG` не вызывается | Нет для публичного `MKFILE`; прямой `MKSG(0)` запрещён |
| E7 | Подозрение на неверный кластер `SRHDRN` | Не является ошибкой: `CLSHL/CLSDE` — поля найденной `ENTRY` | Нет |
| E8 | `SAVE512` не расширяет файл | Для дописывания реализован и проверен API 76 APPEND | Усечение и физические отказы отката |
| E9 | Нет прямого 32-битного seek | Не исправлено | Нужен новый API |
| E10 | Нет атомарной замены файла | Не исправлено | Нужна транзакция или журнал восстановления |
| E11 | Глобальная самомодификация обработчика `NW0` | Восстановление обработчика исправлено; нереентерабельность остаётся | Убрать самомодификацию либо формально сериализовать API |

| E13 | Неверное 32-битное сравнение сектора с размером FAT | Исправлено и пройдено на границах | Нет |
| E14 | `DELFL` удаляет непустой каталог | Не исправлено | Проверка `.`, `..` и содержимого до удаления |
| E15 | Ошибки sector I/O терялись | Распространение ошибки исправлено; транзакционность не реализована | Физические отказы и откат метаданных |
| E16 | Имя 8.3 с пробелом ошибочно становилось SFN | Исправлено для заявленного случая | Расширять таблицу нестандартных имён |

## Границы выполненной проверки

Предыдущий exact-бинарник `f1234d96…` прошёл в патченном Unreal:

- SPC=1, mirrored FAT;
- SPC=1, active FAT1;
- SPC=8, mirrored FAT;
- SPC=8, active FAT1;
- CORE32T: основной результат `stage=8 completed=8`;
- LFN-набор: `completed=10`;
- штатные UI-циклы F5/F8 на SD-ZC и между SD-ZC/Nemo;
- пустой файл, SFN/LFN, 255/256 символов, границы FAT, FSInfo, APPEND и
  искусственный возврат потоковой ошибки.

Текущий exact-бинарник `9c9d4bb5…` отличается от этого runtime-базиса только
двумя операндами выбора страницы `#E0 → #E8`. Для него подтверждены:

- повторяемая сборка;
- совпадение 29 остальных runtime-файлов;
- обратная распаковка `CORE32_EXT`;
- побайтовое совпадение payload внутри HoBeta;
- инъекция в `core32_e8_mirrored.img`;
- C1–C4 для образа;
- успешная ROM-загрузка на реальном устройстве.

Полный CORE32T и совместный сценарий Mounter + CORE32 после переноса страницы
ещё не повторялись. Результаты предыдущего exact-бинарника не выдаются за новый
runtime-прогон.

Образ `core32_e8_mirrored.img` имеет SHA-256
`76d484e199aa1da334d34f248e5f30b2bad82ef95df6f12029f1cc860945f624`.
ROM сначала не нашёл `BOOT.$C` после простой перезаписи старого файла. Рабочая
процедура обновления: удалить старый `BOOT.$C`, записать новый, безопасно
извлечь носитель. После удаления и новой записи загрузка прошла.

## Открытые действия

1. Выполнить совместный runtime-тест Mounter RAM-диска и файловой операции
   CORE32 на странице `#E8`.
2. Исправить E33: не терять `CF` после `CURIT` перед `GIPAG`.
3. Исправить E14/E34/E35: безопасное удаление непустого каталога и однозначные
   ошибки каталожного API.
4. Спроектировать E9/E10: прямой seek и атомарную замену.
5. Убрать остаточную нереентерабельность E11.
6. Перепроектировать конечные ожидания SD-ZC/DSDNGS и проверить их на реальном
   контроллере до возврата в боевой boot.
7. Добавить физические отказные тесты FAT, FSInfo, LFN и APPEND, включая
   транзакционный откат E15.
